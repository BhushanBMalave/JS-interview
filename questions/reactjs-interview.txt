1) Hooks:

1️⃣ useState
What is it?
Used to manage local state in functional components.
When state changes → component re-renders.

Syntax
const [state, setState] = useState(initialValue);

Example
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);

  return (
    <>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </>
  );
}

Key Interview Points
State updates are asynchronous
Never mutate state directly
Can hold any data type

2️⃣ useEffect
What is it?
Used to handle side effects
API calls
Subscriptions
Timers
DOM updates

Syntax
useEffect(() => {
  // side effect
  return () => {
    // cleanup
  };
}, [dependencies]);

Example – API Call
import { useEffect, useState } from "react";

function Users() {
  const [users, setUsers] = useState([]);

  useEffect(() => {
    fetch("/api/users")
      .then(res => res.json())
      .then(data => setUsers(data));
  }, []);

  return <div>{users.length} users</div>;
}

Dependency Variations
Dependency	Behavior
[]        	Runs once (on mount)
[value]	   Runs when value changes
No array	  Runs on every render

3️⃣ useContext
What is it?
Used to avoid prop drilling
Access shared data across components

Example
Create Context
const ThemeContext = React.createContext();

Provide Context
function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Child />
    </ThemeContext.Provider>
  );
}

Consume Context
function Child() {
  const theme = useContext(ThemeContext);
  return <div>{theme}</div>;
}

Interview Notes
Works best with global data
Re-renders all consuming components when value changes.

4️⃣ useReducer
What is it?
Alternative to useState
Best for complex state logic
Similar to Redux pattern

Syntax
const [state, dispatch] = useReducer(reducer, initialState);

Example
function CounterReducer(state, action) {
  switch (action.type) {
    case "increment":
      return { count: state.count + 1 };
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(CounterReducer, { count: 0 });

  return (
    <>
      <p>{state.count}</p>
      <button onClick={() => dispatch({ type: "increment" })}>
        Increment
      </button>
    </>
  );
}

Interview Tip
👉 Use useReducer when:
Multiple state values
Complex updates
State depends on previous state

5️⃣ useMemo
What is it?
Memoizes computed values
Prevents expensive recalculations

Syntax
const memoizedValue = useMemo(() => compute(), [deps]);

Example
const expensiveValue = useMemo(() => {
  return items.reduce((sum, item) => sum + item.price, 0);
}, [items]);

Interview Notes
Improves performance
Do not overuse
Only for expensive calculations

6️⃣ useCallback
What is it?
Memoizes functions
Prevents unnecessary child re-renders

Syntax
const memoizedFn = useCallback(() => {}, [deps]);

Example
const handleClick = useCallback(() => {
  console.log("Clicked");
}, []);

Difference: useMemo vs useCallback
Hook	       Memoizes
useMemo	    Value
useCallback	Function

7️⃣ useRef
What is it?
Holds mutable values
Does not cause re-render
Access DOM elements

Example – DOM Reference
import { useRef } from "react";

function InputFocus() {
  const inputRef = useRef(null);

  return (
    <>
      <input ref={inputRef} />
      <button onClick={() => inputRef.current.focus()}>
        Focus
      </button>
    </>
  );
}

Example – Store Previous Value
const prevCount = useRef(0);

Interview Points
Changes don’t trigger re-render
Useful for timers, DOM, previous state

🔥 Common Interview Questions
Q: Can hooks be used inside loops or conditions?
❌ No — only at the top level

Q: Why hooks were introduced?
✔ To use state & lifecycle features in functional components

Q: Which hook replaces lifecycle methods?
✔ useEffect

2) Higher Order Components (HOC):
🔹 What is a Higher Order Component?
A Higher Order Component is a function that:
Takes a component as input
Returns a new enhanced component

syntax
const EnhancedComponent = higherOrderComponent(OriginalComponent);

➡️ It does not modify the original component
➡️ It wraps the component with extra behavior

🔹 Why use HOC?
Code reusability
Separation of concerns
Avoid repeating the same logic across components
Cleaner, maintainable code

🔹 When to use HOC?
Authentication / Authorization
Logging
Error handling
Injecting props
Conditional rendering

🔹 Basic HOC Example
HOC to add logging
function withLogger(WrappedComponent) {
  return function EnhancedComponent(props) {
    console.log("Props:", props);
    return <WrappedComponent {...props} />;
  };
}

Normal Component
function Profile({ name }) {
  return <h2>Hello {name}</h2>;
}

Enhanced Component
const ProfileWithLogger = withLogger(Profile);

Usage
<ProfileWithLogger name="Bhushan" />

🔍 What happens?
withLogger receives Profile
Returns a new component
Logs props
Renders original component

🔹 Authentication HOC (Very Common Interview Question)
HOC
function withAuth(WrappedComponent) {
  return function AuthComponent(props) {
    const isLoggedIn = true;

    if (!isLoggedIn) {
      return <p>Please login</p>;
    }

    return <WrappedComponent {...props} />;
  };
}

Component
function Dashboard() {
  return <h1>Dashboard</h1>;
}

Wrapped
export default withAuth(Dashboard);

Explanation
If user is authenticated → show component
Else → block access

🔹 HOC with API Data
function withData(WrappedComponent) {
  return function DataComponent(props) {
    const [data, setData] = React.useState([]);

    React.useEffect(() => {
      fetch("/api/data")
        .then(res => res.json())
        .then(setData);
    }, []);

    return <WrappedComponent data={data} {...props} />;
  };
}

🔹 Passing Props Correctly
✔ Always forward props:
<WrappedComponent {...props} />


❌ Wrong:
<WrappedComponent />

🔹 Naming Convention (Best Practice)
withAuth(Component)
withLogger(Component)
withLoading(Component)

🔹 HOC vs Hooks (Interview Favorite)
HOC	              Hooks
Used before Hooks	Introduced in React 16.8
Wrapper component	Function-based logic
Can cause wrapper hell	Cleaner & simpler
Still used in libraries	Preferred for new code

👉 Hooks are preferred today, but HOCs are still important to understand.

🔹 Common Pitfalls
❌ Modifying original component
❌ Forgetting to pass props
❌ Too many nested HOCs

🔹 Real-World Usage
connect() from Redux
withRouter() from React Router (v5)
Error boundaries (older pattern)

🔥 One-Line Interview Answer
A Higher Order Component is a function that takes a component and returns a new component with added functionality, enabling code reuse and separation of concerns.

3) Life Cycle Methods of Components:
Class Components: Traditional React components using ES6 classes.
Mounting: Component is being created and inserted into the DOM.
Updating: Component is being re-rendered as a result of changes.
Unmounting: Component is being removed from the DOM.  deatiled notes for interview with examples

Below are detailed, interview-ready notes on React Life Cycle Methods (Class Components) with clear explanations, execution order, and examples. You can confidently answer both theory + practical questions from this 👇

3️⃣ Life Cycle Methods of Components (Class Components)
🔹 What are Class Components?

Class components are traditional React components written using ES6 classes.
They:
Extend React.Component
Can hold state
Use lifecycle methods
class MyComponent extends React.Component {
  render() {
    return <div>Hello</div>;
  }
}

🔹 What are Lifecycle Methods?
Lifecycle methods are predefined methods that are automatically called by React at different stages of a component’s life:
Mounting
Updating
Unmounting

🔵 1. Mounting Phase
👉 When the component is created and inserted into the DOM

Order of Execution
constructor()
render()
componentDidMount()

1️⃣ constructor(props)
Purpose:
Initialize state
Bind event handlers

constructor(props) {
  super(props);
  this.state = { count: 0 };
}

📌 Interview Notes
Always call super(props)
Do not make API calls here

2️⃣ render()
Purpose:
Returns JSX
Determines UI

render() {
  return <h2>Count: {this.state.count}</h2>;
}

📌 Interview Notes
Must be a pure function
No side effects
Called on every render

3️⃣ componentDidMount()
Purpose:
Runs after component is mounted
Used for:
API calls
Subscriptions
Timers

componentDidMount() {
  console.log("Component Mounted");
}

📌 Interview Notes
Runs only once
Best place for async operations

🟡 2. Updating Phase
👉 Happens when:
State changes (setState)
Props change

Updating Lifecycle Methods
1️⃣ shouldComponentUpdate(nextProps, nextState)
Purpose:
Decide whether component should re-render
shouldComponentUpdate(nextProps, nextState) {
  return nextState.count !== this.state.count;
}

📌 Interview Notes
Improves performance
Returns true or false

2️⃣ render()
Re-renders UI based on new state/props

3️⃣ componentDidUpdate(prevProps, prevState)
Purpose:
Run logic after update
componentDidUpdate(prevProps, prevState) {
  if (prevState.count !== this.state.count) {
    console.log("Count Updated");
  }
}

📌 Interview Notes
Use conditions to avoid infinite loops
Common for dependent API calls

Updating Order
shouldComponentUpdate()
render()
componentDidUpdate()

🔴 3. Unmounting Phase
👉 When component is removed from the DOM
componentWillUnmount()

Purpose:
Cleanup operations:
Clear timers
Remove event listeners
Cancel subscriptions

componentWillUnmount() {
  console.log("Component Unmounted");
}


📌 Interview Notes
Prevents memory leaks
Called once before removal

🧩 Complete Example (All Phases)

class LifeCycleDemo extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
    console.log("Constructor");
  }

  componentDidMount() {
    console.log("Mounted");
  }

  shouldComponentUpdate(nextProps, nextState) {
    return true;
  }

  componentDidUpdate() {
    console.log("Updated");
  }

  componentWillUnmount() {
    console.log("Unmounted");
  }

  render() {
    console.log("Render");
    return (
      <>
        <p>{this.state.count}</p>
        <button onClick={() => this.setState({ count: this.state.count + 1 })}>
          Increment
        </button>
      </>
    );
  }
}

🔁 Lifecycle Methods vs Hooks (Interview Favorite)
Class Lifecycle	         Hook Equivalent
constructor	             useState()
componentDidMount	       useEffect(() => {}, [])
componentDidUpdate	      useEffect(() => {}, [deps])
componentWillUnmount	    useEffect(() => return cleanup)

❌ Deprecated Lifecycle Methods
(Asked occasionally)
componentWillMount
componentWillReceiveProps
componentWillUpdate

➡️ Replaced by hooks and safer alternatives
🔥 Common Interview Questions & Answers
Q: Where should API calls be made?
✔ componentDidMount
Q: How do you prevent unnecessary re-renders?
✔ shouldComponentUpdate
Q: Where do you clean up subscriptions?
✔ componentWillUnmount

✅ One-Line Interview Answer
React lifecycle methods are special methods in class components that allow developers to run code during mounting, updating, and unmounting phases of a component.

4) State management (all about data):
🔹 What is State Management?
State management is how an application stores, updates, and shares data between components so the UI stays in sync with the data.

🔵 1. State vs Props

🔹 State
Internal data managed by the component itself
Can change over time
Triggers re-render

Example – State
function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
    </>
  );
}

Interview Points
✔ Local to component
✔ Mutable via setState / setter
✔ Causes re-render

🔹 Props
External data passed from parent to child
Read-only
Cannot be modified by child

Example – Props
function Child({ name }) {
  return <h3>Hello {name}</h3>;
}

function Parent() {
  return <Child name="Bhushan" />;
}

Interview Points
✔ Passed from parent to child
✔ Immutable
✔ Enables component reuse

🔥 State vs Props (Comparison)
Feature	State	Props
Managed by	Component itself	Parent
Mutability	Mutable	Immutable
Purpose	Store dynamic data	Pass data
Re-render	Yes	Yes

🟡 2. Props Drilling
🔹 What is Props Drilling?
Props drilling occurs when data is passed through multiple intermediate components that don’t need the data, just to reach a deeply nested child.

Example – Props Drilling Problem
function App() {
  const user = "Bhushan";
  return <Parent user={user} />;
}

function Parent({ user }) {
  return <Child user={user} />;
}

function Child({ user }) {
  return <GrandChild user={user} />;
}

function GrandChild({ user }) {
  return <h1>{user}</h1>;
}

Problems
❌ Unnecessary props
❌ Hard to maintain
❌ Tightly coupled components

🟢 3. Context API (Solution to Props Drilling)
🔹 What is Context?
Context allows you to share data globally without passing props manually at every level.

🔹 When to Use Context?
Theme (dark/light)
User authentication
Language settings
Global app config

🔹 Context Example (Step by Step)
1️⃣ Create Context
const UserContext = React.createContext();

2️⃣ Provide Context
function App() {
  const user = "Bhushan";

  return (
    <UserContext.Provider value={user}>
      <Parent />
    </UserContext.Provider>
  );
}

3️⃣ Consume Context
function GrandChild() {
  const user = React.useContext(UserContext);
  return <h1>{user}</h1>;
}

🔹 Without Props Drilling
No need to pass props through Parent → Child
Direct access to data

🔥 Props Drilling vs Context
Feature	        Props       Drilling	Context
Data passing	   Manual	     Automatic
Maintainability	 Poor       	Better
Best for	       Small apps	 Medium/large apps
Performance	     Better	     Can cause re-renders

🧠 Real-World Example: Theme Context
const ThemeContext = React.createContext();

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Navbar />
    </ThemeContext.Provider>
  );
}

function Navbar() {
  return <ThemeButton />;
}

function ThemeButton() {
  const theme = React.useContext(ThemeContext);
  return <button>{theme}</button>;
}

⚠️ Context Performance Tip (Interview Bonus)
Context updates re-render all consumers
Avoid using context for frequently changing data
Combine with useMemo if needed

🔁 Context vs Redux (Quick Interview Answer)
Context → Data sharing
Redux → State management library
Context is not a Redux replacement

🔥 Common Interview Questions
Q: Can a child modify props?
❌ No
Q: Why props are immutable?
✔ To ensure predictable data flow
Q: Does Context replace Redux?
❌ No, it only solves prop drilling

✅ One-Line Interview Answer
State management in React deals with how data is stored, passed, and shared across components using state, props, props drilling, and Context API.

5) Redux
🔹 What is Redux?
Redux is a predictable state container for JavaScript applications that provides a single source of truth for the entire app.
All application state is stored in one global store.

🔹 Why Redux?
Redux solves problems like:
Props drilling
Inconsistent state
Difficult debugging
Managing shared state across many components

🔹 When Should You Use Redux?
✅ Large applications
✅ Multiple components need the same data
✅ Complex state transitions
❌ Small apps (Context is enough)

🔵 Core Redux Concepts
1️⃣ Store
Holds the entire app state
Only one store per app

const store = createStore(reducer);

2️⃣ Action
Plain JavaScript object
Describes what happened

{
  type: "INCREMENT",
  payload: 1
}

📌 Actions are read-only

3️⃣ Reducer
A pure function
Takes current state + action
Returns new state

function counterReducer(state = { count: 0 }, action) {
  switch (action.type) {
    case "INCREMENT":
      return { count: state.count + action.payload };
    default:
      return state;
  }
}

4️⃣ Dispatch
Sends an action to the store

store.dispatch({ type: "INCREMENT", payload: 1 });

5️⃣ Selector
Reads data from the store

const count = useSelector(state => state.counter.count);

🔁 How Redux Works (Data Flow)
Component → Dispatch Action → Reducer → Store → UI Update

UI triggers an action
Action is dispatched
Reducer calculates new state
Store updates
UI re-renders

➡️ One-way data flow
🟡 Redux Example (Without Toolkit)

import { createStore } from "redux";

const reducer = (state = { count: 0 }, action) => {
  switch (action.type) {
    case "INC":
      return { count: state.count + 1 };
    default:
      return state;
  }
};

const store = createStore(reducer);

❌ Too much boilerplate
❌ Manual immutability
❌ Extra setup

🟢 Redux Toolkit (RTK)
🔹 What is Redux Toolkit?
Redux Toolkit is the official, recommended way to write Redux logic.

✔ Less boilerplate
✔ Built-in best practices
✔ Uses Immer internally
✔ Easy async handling

🔹 RTK Core APIs
API	Purpose
configureStore	Create store
createSlice	Create reducer + actions
createAsyncThunk	Handle async logic

🧩 RTK Example (Step-by-Step)
1️⃣ Create Slice
import { createSlice } from "@reduxjs/toolkit";

const counterSlice = createSlice({
  name: "counter",
  initialState: { count: 0 },
  reducers: {
    increment(state) {
      state.count += 1; // mutating (safe via Immer)
    },
    decrement(state) {
      state.count -= 1;
    }
  }
});

export const { increment, decrement } = counterSlice.actions;
export default counterSlice.reducer;

2️⃣ Configure Store
import { configureStore } from "@reduxjs/toolkit";
import counterReducer from "./counterSlice";

export const store = configureStore({
  reducer: {
    counter: counterReducer
  }
});

3️⃣ Provide Store
import { Provider } from "react-redux";
import { store } from "./store";

<Provider store={store}>
  <App />
</Provider>

4️⃣ Use in Component
import { useDispatch, useSelector } from "react-redux";
import { increment } from "./counterSlice";

function Counter() {
  const count = useSelector(state => state.counter.count);
  const dispatch = useDispatch();

  return (
    <>
      <p>{count}</p>
      <button onClick={() => dispatch(increment())}>+</button>
    </>
  );
}

⚡ Async Logic with RTK (createAsyncThunk)
export const fetchUsers = createAsyncThunk(
  "users/fetch",
  async () => {
    const res = await fetch("/api/users");
    return res.json();
  }
);


Used inside slice:

extraReducers: builder => {
  builder
    .addCase(fetchUsers.pending, state => {
      state.loading = true;
    })
    .addCase(fetchUsers.fulfilled, (state, action) => {
      state.loading = false;
      state.users = action.payload;
    });
}

🔥 Redux vs Context (Interview Must-Know)
Redux	                Context
State management	     Data sharing
Predictable updates	  Simple
Middleware support	   No middleware
Best for large apps	  Small/medium apps

❓ Common Interview Questions
Q: Why reducers must be pure?
✔ To ensure predictable state updates

Q: Why single source of truth?
✔ Easier debugging & consistency

Q: Is Redux synchronous?
✔ Core Redux is synchronous; async handled via middleware/RTK

✅ One-Line Interview Answer
Redux is a centralized state management library that uses actions and reducers to manage application state in a predictable, one-way data flow, and Redux Toolkit simplifies its setup and usage.

6) Custom Hooks:
🔹 What are Custom Hooks?
A Custom Hook is a JavaScript function:
Whose name starts with use
That uses other hooks internally
Returns stateful logic (state, functions, values)

function useSomething() {
  // hook logic
}

👉 Custom Hooks do not return JSX, they return logic.

🔹 Why Custom Hooks?
Custom Hooks solve:
Repeated logic across components
Messy components with too much logic
Poor readability and maintainability

✔ Better code reuse
✔ Clean components
✔ Separation of concerns

🔹 When to Use Custom Hooks?
Use them when:
Same useEffect + useState logic is repeated
Logic is shared across multiple components
Component becomes too large
Business logic needs isolation

🔹 Rules of Custom Hooks (Interview Must-Know)
Name must start with use
Call hooks only at top level
Can call other hooks
Cannot be called conditionally

🔵 Simple Custom Hook Example
❌ Without Custom Hook
function ComponentA() {
  const [count, setCount] = React.useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

function ComponentB() {
  const [count, setCount] = React.useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

✅ With Custom Hook
function useCounter(initialValue = 0) {
  const [count, setCount] = React.useState(initialValue);

  const increment = () => setCount(c => c + 1);

  return { count, increment };
}

Usage
function Counter() {
  const { count, increment } = useCounter(0);
  return <button onClick={increment}>{count}</button>;
}

🟡 Custom Hook with useEffect (API Logic)
function useFetch(url) {
  const [data, setData] = React.useState(null);
  const [loading, setLoading] = React.useState(true);

  React.useEffect(() => {
    fetch(url)
      .then(res => res.json())
      .then(data => {
        setData(data);
        setLoading(false);
      });
  }, [url]);

  return { data, loading };
}

Usage
function Users() {
  const { data, loading } = useFetch("/api/users");

  if (loading) return <p>Loading...</p>;
  return <div>{data.length} users</div>;
}

🟢 Custom Hook for Window Size
function useWindowSize() {
  const [width, setWidth] = React.useState(window.innerWidth);

  React.useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener("resize", handleResize);

    return () => window.removeEventListener("resize", handleResize);
  }, []);

  return width;
}

🔥 Custom Hook vs HOC vs Render Props
Pattern	         Purpose	         Complexity
Custom Hook	Reuse logic	Simple
HOC	Enhance components	Medium
Render Props	Share render logic	Complex

👉 Custom Hooks are preferred today

🧠 Real-World Custom Hooks
useAuth
useDebounce
usePrevious
useLocalStorage

Example: useLocalStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = React.useState(() => {
    return JSON.parse(localStorage.getItem(key)) ?? initialValue;
  });

  React.useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

⚠️ Common Mistakes (Interview Traps)
❌ Returning JSX
❌ Calling hook conditionally
❌ Forgetting dependency array

🔥 Common Interview Questions
Q: Can custom hooks use other hooks?
✔ Yes

Q: Do custom hooks share state?
❌ No, logic is shared, not state

Q: Why must hook names start with use?
✔ React relies on naming convention for rules enforcement

✅ One-Line Interview Answer
Custom Hooks are reusable functions that encapsulate React hook logic to improve code reusability, readability, and separation of concerns.

7) Lazy Loading:
🔹 What is Lazy Loading?
Lazy loading is a performance optimization technique where components or resources are loaded only when needed, instead of loading everything at once.

✔ Faster initial load
✔ Smaller bundle size
✔ Better user experience

🔵 1. Code Splitting
🔹 What is Code Splitting?
Code splitting means breaking a large JavaScript bundle into smaller chunks so that only required code is loaded.

Without Code Splitting
Entire app JS loads at once
Slow initial page load

With Code Splitting
Only essential code loads first
Remaining code loads on demand

🔹 React Support for Code Splitting
React supports code splitting using:
React.lazy()
Dynamic import()

Example – Without Lazy Loading
import Dashboard from "./Dashboard";

function App() {
  return <Dashboard />;
}
❌ Dashboard loaded even if user never visits it

Example – With Code Splitting
const Dashboard = React.lazy(() => import("./Dashboard"));
✔ Dashboard loaded only when rendered

🟡 2. Chunking
🔹 What is Chunking?
Chunking is the result of code splitting, where:
App code is divided into chunks
Each chunk is loaded independently

📌 Bundlers like Webpack / Vite handle chunk creation automatically.

Example – Route-Based Chunking
import { Routes, Route } from "react-router-dom";

const Home = React.lazy(() => import("./Home"));
const About = React.lazy(() => import("./About"));

function App() {
  return (
    <React.Suspense fallback={<p>Loading...</p>}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
      </Routes>
    </React.Suspense>
  );
}

✔ Each route loads its own chunk
✔ Better performance for large apps

🟢 3. React.Suspense
🔹 What is Suspense?
Suspense allows React to:
Pause rendering
Show a fallback UI
Resume once lazy component is loaded

Syntax
<Suspense fallback={<Loading />}>
  <LazyComponent />
</Suspense>

Example – Component Level
const Profile = React.lazy(() => import("./Profile"));

function App() {
  return (
    <React.Suspense fallback={<div>Loading Profile...</div>}>
      <Profile />
    </React.Suspense>
  );
}

8) Virtual DOM:
🔹 What is the Virtual DOM?
The Virtual DOM (VDOM) is a lightweight JavaScript representation of the real DOM.

It is an in-memory tree
Describes what the UI should look like
Faster to update than the real DOM

const vdom = {
  type: "h1",
  props: { children: "Hello" }
};


👉 React never directly updates the real DOM first.

🔵 Rendering in React (High Level)
What is a Render?
A render is the process of:
Executing component functions / render methods
Creating a new Virtual DOM tree
Comparing it with the previous one
Updating only the necessary parts of the real DOM

🔁 Render Flow (Interview Explanation)
State / Props change
        ↓
Component re-renders
        ↓
New Virtual DOM created
        ↓
Diffing (Reconciliation)
        ↓
Minimal DOM updates (Commit phase)

🟡 Reconciliation Algorithm
🔹 What is Reconciliation?
Reconciliation is the process where React:
Compares old VDOM with new VDOM
Figures out what changed
Updates only those changes in the real DOM

👉 This makes React fast and efficient.

🔹 Key Assumptions of Diff Algorithm
React uses heuristics (rules) to optimize diffing:

1️⃣ Elements with different types → replace entire subtree
2️⃣ Elements with same type → update only changed props
3️⃣ Lists use keys to identify elements

Example – Same Element Type
<h1 className="red">Hello</h1>
<h1 className="blue">Hello</h1>

✔ Only className is updated
❌ DOM node is not recreated

Example – Different Element Type
<h1>Hello</h1>
<p>Hello</p>

❌ h1 destroyed
✔ p created

Example – Lists & Keys
{items.map(item => (
  <li key={item.id}>{item.name}</li>
))}

✔ Keys help React identify:
Insertions
Deletions
Reorering

❌ Using index as key → bad for dynamic lists

🟢 Diff Algorithm (Simplified)
🔹 What is Diffing?
Diffing is the comparison process between:
Previous Virtual DOM
Current Virtual DOM
Goal: Find minimal changes

Example
<ul>
  <li>A</li>
  <li>B</li>
</ul>

Update:
<ul>
  <li>A</li>
  <li>C</li>
</ul>

✔ React updates only text B → C
❌ Does not recreate <ul>

🔴 React Fiber
🔹 What is React Fiber?
React Fiber is a complete reimplementation of React’s core reconciliation algorithm (introduced in React 16).

Why Fiber?
Old React:
Synchronous rendering
UI could block on heavy updates

Fiber:
Breaks rendering into units of work
Can pause, resume, or cancel work
Prioritizes updates

🔹 Benefits of React Fiber
✔ Smooth animations
✔ Better responsiveness
✔ Concurrent rendering
✔ Error boundaries support

🔹 Fiber Concepts (Interview Level)
Work Units: Small pieces of render work
Priority: Urgent updates first
Time Slicing: Pause work to keep UI responsive

🔵 How Does render() Work?
🔹 Render in Class Component
render() {
  return <h1>{this.state.count}</h1>;
}

Called when:
State changes
Props change
Should be pure
Does not modify DOM directly

🔹 Render in Functional Component
function Counter() {
  return <h1>Counter</h1>;
}

Same process:
✔ Function executes
✔ VDOM created → compared → committed

⚠️ Important Interview Clarifications
❓ Does render mean DOM update?
❌ No
✔ Render = creating Virtual DOM
✔ Commit phase = DOM update

🧠 Render vs Commit Phase
Phase	Description
Render	Calculate UI (VDOM)
Commit	Apply changes to real DOM
🔥 Common Interview Questions

Q: Is Virtual DOM faster than Real DOM?
✔ Updating VDOM is faster; DOM updates are minimized

Q: Why are keys important?
✔ Helps React identify elements during diffing

Q: Does state change always cause re-render?
✔ Yes, unless optimized (memo, shouldComponentUpdate)

✅ One-Line Interview Answer

9) SSR vs CSR (important):
🔹 What is Rendering?
Rendering is the process of converting components into HTML that the browser can display.

🔵 Client-Side Rendering (CSR)
🔹 What is CSR?
In Client-Side Rendering, the browser:
Downloads a minimal HTML file
Downloads JavaScript
JavaScript renders the UI in the browser

👉 React SPA (Create React App) is CSR by default.

CSR Flow
Request → index.html
        → JS bundle
        → React renders UI
        → Content visible

CSR Example
<!-- index.html -->
<div id="root"></div>

ReactDOM.createRoot(document.getElementById("root")).render(<App />);

Pros of CSR
✔ Fast navigation after load
✔ Rich interactivity
✔ Simple architecture

Cons of CSR
❌ Poor SEO (empty HTML initially)
❌ Slower first content paint
❌ JS-heavy initial load

🔴 Server-Side Rendering (SSR)
🔹 What is SSR?
In Server-Side Rendering, the server:
Generates HTML for each request
Sends fully rendered HTML to the browser

👉 Common in Next.js

SSR Flow
Request → Server renders HTML
        → HTML sent to browser
        → JS hydrates React

SSR Example (Next.js)
export async function getServerSideProps() {
  const data = await fetch("https://api.example.com/posts").then(res => res.json());
  return { props: { data } };
}

export default function Page({ data }) {
  return <h1>{data.title}</h1>;
}

Pros of SSR
✔ Better SEO
✔ Faster initial page load
✔ Content visible immediately

Cons of SSR
❌ Higher server load
❌ Slower navigation after initial load
❌ More complex setup

🟡 SEO & Performance Comparison
SEO
CSR → Search engines see empty HTML
SSR → Search engines see full content
✔ SSR is SEO-friendly

Performance
Metric	                   CSR	                             SSR
Initial load	             Slower	                           Faster
TTI (Time to Interactive)	 Faster	                           Slightly slower
Subsequent navigation	     Faster                          	 Fast

🔁 CSR vs SSR (Comparison Table)
Feature	                      CSR	                          SSR
Rendering location	          Browser	                      Server
Initial HTML	                Empty	                        Fully rendered
SEO	                          Poor	                        Excellent
First load	                  Slower	                      Faster
Complexity	                  Simple	                      Complex
Server cost	                  Low	                          Higher

🧠 Hydration (Important Interview Term)
🔹 What is Hydration?
Hydration is the process where:
React attaches event listeners
Converts server-rendered HTML into a fully interactive app

HTML (from server) + JS → Interactive React App

🟢 When to Use What?
Use CSR when:
Internal dashboards
Admin panels
Apps with low SEO needs
Use SSR when:

Marketing pages
Blogs
E-commerce
SEO-critical apps

🔥 CSR vs SSR vs SSG (Bonus)
Type	Description
CSR	Render in browser
SSR	Render on every request
SSG	Render at build time

❓ Common Interview Questions
Q: Is SSR always faster?
❌ No, only for initial load

Q: Does SSR eliminate JS?
❌ No, JS is still needed for hydration

Q: Can SSR work without hydration?
❌ No interactivity without JS

✅ One-Line Interview Answer
CSR renders content in the browser using JavaScript, while SSR renders HTML on the server, improving SEO and initial page load performance.

12) Async Tasks:
Async tasks are operations that don’t block the UI thread, such as:

🔵 API Calls (Fetching Data)
Common Ways

axios

Basic Example
import { useEffect, useState } from "react";

function Users() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch("/api/users")
      .then(res => res.json())
      .then(data => {
        setUsers(data);
        setLoading(false);
      })
      .catch(err => {
        console.error(err);
        setLoading(false);
      });
  }, []);

  if (loading) return <p>Loading...</p>;

  return users.map(user => <p key={user.id}>{user.name}</p>);
}


📌 Interview points:
API calls are side effects
Should be placed inside useEffect
Handle loading & error states

🔵 useEffect In-Depth (VERY IMPORTANT)
What is useEffect?
Runs side effects after render:
API calls
Subscriptions
Timers
DOM updates

Syntax
useEffect(() => {
  // side effect
  return () => {
    // cleanup
  };
}, [dependencies]);

Dependency Array Cases
1️⃣ No dependency array
useEffect(() => {
  console.log("Runs on every render");
});

2️⃣ Empty array []
useEffect(() => {
  console.log("Runs once (on mount)");
}, []);

3️⃣ With dependencies
useEffect(() => {
  console.log("Runs when count changes");
}, [count]);

❗ Async Inside useEffect (Correct Pattern)

❌ Wrong

useEffect(async () => {
  const data = await fetchData();
}, []);


✔ Correct
useEffect(() => {
  const loadData = async () => {
    const data = await fetchData();
  };
  loadData();
}, []);

📌 Reason: useEffect callback must return cleanup or nothing, not a Promise.

🔴 Cleanup Function (Interview Favorite)
Example – Event Listener

  window.addEventListener("resize", handleResize);

  return () => {
    window.removeEventListener("resize", handleResize);
  };
}, []);

🔴 Avoiding Memory Leaks (IMPORTANT)
Problem
Component unmounts before async completes → state update error

Solution
useEffect(() => {
  let isMounted = true;

  fetchData().then(data => {
    if (isMounted) setState(data);
  });

  return () => {
    isMounted = false;
  };
}, []);

🔴 Virtualization (Large Lists)
🔹 What is Virtualization?
Virtualization is a performance optimization technique where:
Only visible items in a list are rendered
Off-screen items are not mounted in the DOM

👉 Prevents UI slowdown when rendering large datasets

🔹 Why is Virtualization Needed?
Problem (Without Virtualization)
{items.map(item => <Row key={item.id} />)}

❌ If items.length = 10,000

DOM becomes heavy
Slow rendering
High memory usage
Solution

Render only what the user sees + a small buffer.

🔵 react-window (Most Common Library)
🔹 What is react-window?
A lightweight library by Brian Vaughn (React team) that:
Efficiently renders large lists
Supports fixed & variable item sizes
Very small bundle size

🔹 Installation
npm install react-window

🟢 Fixed Size List Example
import { FixedSizeList as List } from "react-window";

const Row = ({ index, style }) => (
  <div style={style}>
    Row {index}
  </div>
);

function VirtualizedList() {
  return (
    <List
      height={400}
      itemCount={10000}
      itemSize={35}
      width="100%"
    >
      {Row}
    </List>
  );
}

🔹 Explanation (Interview-Friendly)
height → visible container height
itemCount → total number of items
itemSize → fixed height of each row
style → must be applied for positioning

📌 Only ~10–15 rows render at a time instead of 10,000

🟣 Variable Size List Example
import { VariableSizeList as List } from "react-window";

const getItemSize = index => index % 2 === 0 ? 50 : 80;

<List
  height={400}
  itemCount={1000}
  itemSize={getItemSize}
  width="100%"
>
  {Row}
</List>

🔵 Grid Virtualization
import { FixedSizeGrid as Grid } from "react-window";

<Grid
  columnCount={5}
  columnWidth={100}
  height={300}
  rowCount={1000}
  rowHeight={35}
  width={500}
>
  {({ columnIndex, rowIndex, style }) => (
    <div style={style}>
      {rowIndex}, {columnIndex}
    </div>
  )}
</Grid>

📌 Used for tables, dashboards

🔴 Key Rules (Very Important)
✔ Always apply style prop
✔ Use stable keys when possible
✔ Avoid heavy components inside rows
✔ Combine with React.memo

In React, writing HTML without JSX means using React.createElement() directly.
This is a common interview question to check if you understand what JSX actually does behind the scenes.

✅ 1. What is JSX?
JSX is just syntactic sugar for React.createElement().
This:
const element = <h1>Hello World</h1>;

Is converted (by Babel) into:
const element = React.createElement('h1', null, 'Hello World');

✅ 2. Writing HTML in React Without JSX
Instead of writing HTML-like syntax, we use:
React.createElement(type, props, ...children)

🔹 Example 1 – Simple Element
import React from 'react';
import ReactDOM from 'react-dom/client';

const element = React.createElement(
  'h1',
  null,
  'Hello World'
);

const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(element);

🔹 Example 2 – With Props (Attributes)
JSX version:
<a href="https://google.com">Google</a>

Without JSX:
const element = React.createElement(
  'a',
  { href: 'https://google.com' },
  'Google'
);

🔹 Example 3 – Nested Elements
JSX:
<div>
  <h1>Hello</h1>
  <p>Welcome</p>
</div>

Without JSX:
const element = React.createElement(
  'div',
  null,
  React.createElement('h1', null, 'Hello'),
  React.createElement('p', null, 'Welcome')
);

✅ 3. Functional Component Without JSX

JSX:
function App() {
  return <h1>Hello React</h1>;
}

Without JSX:
function App() {
  return React.createElement('h1', null, 'Hello React');
}

✅ 4. Interview One-Line Answer
How do you write HTML in React without JSX?
👉 “In React, we can write HTML without JSX by using React.createElement(). JSX is just syntactic sugar that gets transpiled into React.createElement() calls.”

Controlled vs Uncontrolled Components (React Forms)

Forms in React can be handled in two ways:

1️⃣ Controlled Components
2️⃣ Uncontrolled Components

The difference is where the form data is stored.

🔵 Controlled Components
🔹 What?

A controlled component is a form element whose value is controlled by React state.

👉 React state is the single source of truth.

Example
import { useState } from "react";

function Form() {
  const [name, setName] = useState("");

  const handleChange = (e) => {
    setName(e.target.value);
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(name);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={name} onChange={handleChange} />
      <button type="submit">Submit</button>
    </form>
  );
}
🔹 How It Works

User types → onChange triggers → setState updates → UI re-renders

Flow:

Input → onChange → React State → UI Update
✅ Advantages

✔ Full control over form data
✔ Easy validation
✔ Predictable behavior
✔ Good for complex forms

❌ Disadvantages

❌ More code
❌ Re-render on every keystroke

🔴 Uncontrolled Components
🔹 What?

Form data is handled by the DOM itself, not React state.

👉 Access values using refs.

Example
import { useRef } from "react";

function Form() {
  const inputRef = useRef();

  const handleSubmit = (e) => {
    e.preventDefault();
    console.log(inputRef.current.value);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input ref={inputRef} />
      <button type="submit">Submit</button>
    </form>
  );
}
🔹 How It Works

User types → DOM stores value → React reads using ref

Flow:

Input → DOM → useRef → Access Value
✅ Advantages

✔ Less code
✔ Better performance for large forms
✔ Works well with non-React libraries

❌ Disadvantages

❌ Harder validation
❌ Less control
❌ Not ideal for complex forms

🟡 Controlled vs Uncontrolled (Comparison)
Feature	Controlled	Uncontrolled
Data stored in	React State	DOM
Value access	state	ref
Validation	Easy	Hard
Performance	More re-renders	Faster
Use case	Complex forms	Simple forms
🔵 Example – Controlled Input Validation
const [email, setEmail] = useState("");

const isValid = email.includes("@");

<input value={email} onChange={(e) => setEmail(e.target.value)} />

{!isValid && <p>Invalid email</p>}

📌 Very easy to validate

🔴 Example – Uncontrolled Default Value
<input defaultValue="Bhushan" ref={inputRef} />
🧠 Interview Hack – Best Answer

Q: What is the difference between controlled and uncontrolled components?

Controlled components store form data in React state and update via onChange handlers, while uncontrolled components rely on the DOM to manage form state and access values using refs.
